# 罗莱 PDCA 智能化管理系统 — 产品需求文档

**版本：** v2.1
**日期：** 2026-04-02
**状态：** 已确认

---

## Problem Statement

罗莱生活拥有全国近千家线下门店，营销总监需要实时掌握各门店销售进度、及时发现滞后门店、推动各级管理者执行跟进动作。

目前依赖 Excel 手工汇报，存在以下核心问题：

- **数据滞后**：汇报路径长（店员→督导→片区经理→总监），信息传递慢，问题发现不及时
- **执行断层**：目标下发后缺乏过程跟踪机制，只看结果不看过程
- **激励缺失**：优秀表现无展示渠道，店员之间缺乏正向竞争氛围
- **经验孤岛**：优秀销售经验停留在个人，无法系统化传递给团队

---

## Solution

**罗莱 PDCA 智能化管理系统** — 一套以 PDCA 管理循环为核心的线上化销售跟进平台。

- **P（计划）**：月度目标由总监逐级下发至门店/店员，形成完整目标链，每月 1 日前完成级联
- **D（执行）**：店员每日在线填报销售数据，系统自动汇聚到各层级看板
- **C（检查）**：系统自动计算达成状态，预警滞后门店，飞书推送汇总通知
- **A（总结）**：AI 自动生成按督导维度的周复盘报告，督导可编辑后发布

同时具备排行榜、经验推送功能，将管理平台升级为兼具激励与经验传播的**销售作战平台**。

## User Stories

### 认证与组织

1. 作为**任意角色用户**，我希望通过飞书账号扫码登录，无需记忆独立系统账号密码，降低使用门槛
2. 作为**系统管理员**，我希望系统自动从飞书通讯录同步组织架构，自动建立总监→片区经理→督导→店员的层级关系，无需手动录入
3. 作为**系统管理员**，我希望用户的系统角色根据其飞书通讯录中的职位自动分配，无需逐一手动配置
4. 作为**任意角色用户**，我希望退出飞书后系统会话自动失效，保证数据安全
5. 作为**系统管理员**，我希望当飞书通讯录发生人员新增、离职、调岗变动时，系统能自动同步更新组织架构和角色
6. 作为**系统管理员**，我希望能手动触发一次飞书组织架构全量同步，以应对批量调整等特殊情况

### P · 目标与计划

7. 作为**督导**，我希望在每月初于系统中自行设定本督导区域的月度销售总目标
8. 作为**督导**，我希望收到飞书通知后进入系统，将督导目标拆分下发至管辖的每个门店，再由门店拆分至每位店员
9. 作为**督导**，我希望在拆分目标时，系统实时校验各门店目标之和是否满足我的督导目标（可超发，不可少发），防止漏分
10. 作为**督导**，我希望在拆分门店目标至店员时，系统实时校验各店员目标之和是否等于该门店目标，确保目标闭环
11. 作为**店员**，我希望收到飞书通知告知我本月个人目标金额
12. 作为**系统**，当督导在每月 1 日前仍未完成目标向门店/店员拆分时，自动通过飞书发送提醒，督促其完成分配
13. 作为**督导**，我希望能修改已下发给门店/店员的目标，修改后自动触发飞书重新通知相关店员
14. 作为**督导**，我希望系统根据月度目标自动推算出周参考目标（月目标 ÷ 当月周数），供我参考分解进度
15. 作为**任意管理角色**，我希望在拆分界面实时看到「已分配 / 待分配」的差额，防止超发过多或分配遗漏

### D · 落地与执行

21. 作为**店员**，我希望每天在系统中填写当日实际销售额，记录我的执行结果
22. 作为**店员**，当我当日销售额未达成日目标时，系统强制要求我填写未达成原因（自由文本），否则无法提交，确保过程留痕
23. 作为**店员**，当我当日销售额达成或超过日目标时，我可以选填成功总结，分享我的销售心得（非强制）
24. 作为**店员**，我希望在每天 19:00 收到飞书提醒，告知我当日日报尚未提交，给我充足时间填写
25. 作为**店员**，我希望日报提交截止时间为每天 20:00；20:00 后若仍未提交，系统自动将该日状态标记为「未填报」
26. 作为**店员**，我在系统中只能填写当天的日报，不能补填过去未填报的日期，确保数据真实性
27. 作为**督导**，我希望每天能看到我管辖范围内今日哪些门店/店员已填报、哪些未填报，方便督促跟进
28. 作为**督导**，我希望能查看每个店员/门店的历史日报记录，了解过往执行情况
29. 作为**督导**，当我管辖范围内有店员在 20:00 后仍未填报时，收到 1 条飞书汇总通知（而非多条逐一通知），告知未填报名单

### C · 状态与监控

30. 作为**系统**，在店员提交日报后自动计算其当日目标完成率，并以红/绿状态标记：绿（≥100%，达成目标），红（<100%，未达成目标）
31. 作为**系统**，自动汇总各门店、各督导、各片区在当月维度的累计完成率，并同步更新红/绿状态
32. 作为**督导**，我希望在看板中直观看到我管辖门店的红/绿状态分布，一眼识别哪些门店需要重点跟进
33. 作为**督导**，我希望能点击具体的红色预警门店或店员，快速查看其近期日报明细和未达成原因
34. 作为**系统**，当某店员连续 2 个自然日未提交日报时，自动向其督导发送飞书预警通知（1 条汇总，按店汇总未填报名单）
35. 作为**系统**，当某门店当前月累计完成率低于 60% 时，自动向对应督导发送飞书预警通知

### 排行榜

38. 作为**任意角色用户**，我希望在排行榜页看到本周全国店员 TOP10 排名（默认按目标完成率排序）
39. 作为**任意角色用户**，我希望在排行榜页看到本周全国门店 TOP10 排名
40. 作为**任意角色用户**，我希望排行榜支持两个维度切换：目标完成率 / 销售额绝对值，满足不同视角的对比需求
41. 作为**任意角色用户**，我希望当两个参与者完成率相同时，系统以销售额绝对值作为破平规则，确保排名唯一
42. 作为**任意角色用户**，我希望看到本周销售冠军的风采照片、姓名、所属门店、销售额和完成率，营造荣誉感
43. 作为**督导**，我希望为本周冠军上传风采照片，并设置该照片作为该店员的长期个人档案照，后续自动沿用直到有新冠军或手动更新
44. 作为**督导**，我能查看我管辖范围内所有门店和店员的完整排名数据，非我管辖范围的数据不可见
45. 作为**系统**，当月中新入职的店员（加入飞书通讯录后首次同步）不参与当月已进行周次的排行榜，避免因基数小导致完成率虚高
46. 作为**店员**，我希望能看到全国 TOP10 排行榜，激励自己对标优秀同事

### A · 复盘与总结

47. 作为**系统**，每周一自动基于上周（周一至周日）数据，为每位督导各自生成一份 AI 周复盘报告，内容包含：累计完成率、环比上周变化（含具体百分比数字）、未达成原因分类汇总、TOP 表现门店/店员亮点
48. 作为**督导**，我希望在每周一收到飞书通知，告知本周 AI 复盘报告已生成，可进入系统查看和编辑
49. 作为**督导**，我希望能在系统内对 AI 生成的复盘内容进行编辑，加入我个人的管理意见和下周行动计划
50. 作为**督导**，我希望编辑完成后确认发布，将复盘报告推送给我管辖的所有店员，让团队了解上周整体表现
51. 作为**督导**，发布后若发现内容需要修正，仍可重新编辑并再次发布，系统记录最后一次发布版本
52. 作为**店员**，我希望收到督导发布复盘的飞书通知后，进入系统查看本周团队复盘报告

### 经验推送

55. 作为**系统管理员**，我希望能浏览所有店员填写的成功总结，从中挑选优质内容推送到首页经验区
56. 作为**系统管理员**，推送的经验内容为纯文字，并自动附上来源店员姓名和所属门店，增强可信度
57. 作为**系统管理员**，首页经验区最多展示最新 10 条，当我推送第 11 条时，最旧的一条自动移除（FIFO）
58. 作为**任意角色用户**，我希望在系统首页看到最新的优秀销售经验推送，学习他人的成功做法
59. 作为**店员**，我希望能看到经验作者的姓名和门店，了解成功案例的真实背景

### 数据看板

53. 作为**督导**，我希望在看板看到我管辖门店的当月汇总、预警门店列表及未填报名单
54. 作为**督导**，我希望能查看最近 3 个月我管辖范围的月度完成率趋势
55. 作为**店员**，我希望在看板看到我个人的当月累计销售额、目标完成率、本周排名及今日填报状态
56. 作为**任意角色用户**，我希望看板数据实时反映最新填报情况，无需手动刷新等待

### 数据归档

57. 作为**系统**，每年年底自动将上一年度的所有日报、周复盘、目标记录归档，历史数据仍可查阅但不影响当年运营
58. 作为**任意管理角色**，我希望能查阅已归档的历史年度数据，以便跨年度对比和复盘

## Implementation Decisions

### 角色体系（一期 3 种，二期扩展至 5 种）

| 角色 | 英文标识 | 核心权限 | 期次 |
|------|----------|----------|------|
| 门店督导 | `SUPERVISOR` | 管辖门店全量数据；自设月度目标并拆分至门店/店员；上传冠军风采照；编辑发布周复盘 | 一期 |
| 店员 | `STAFF` | 个人数据 + 全国 TOP10；填写日报 | 一期 |
| 系统管理员 | `ADMIN` | 系统配置；飞书组织同步；经验推送管理 | 一期 |
| 营销总监 | `DIRECTOR` | 全局数据只读；设定全国总目标并下发片区 | 二期 |
| 片区经理 | `REGIONAL_MANAGER` | 片区数据只读；设定督导目标并下发 | 二期 |

> 一期数据库 schema 中保留 `DIRECTOR` / `REGIONAL_MANAGER` 枚举值，以便二期扩展时无需迁移。登录后若角色为这两者，系统展示「功能即将开放」占位页。

### 业务规则汇总

**目标级联规则（一期）：**
- 级联链：督导（自设）→ 门店 → 店员
- 督导自行设定本区域月度目标，无需上级下发
- 每月 1 日为督导完成拆分的截止基准；未完成时飞书自动提醒
- 上级可超发目标给下级（下级合计可超过上级总量），但不可少发（下级合计不得低于督导月度目标）
- 门店目标 = 门店内所有店员目标之和（严格相等）
- 目标可修改，修改后触发受影响层级重新通知

**日报规则：**
- 必须在线提交，不支持离线暂存
- 截止时间 20:00，逾期自动标记「未填报」
- 未达成日目标时，未达成原因为必填字段
- 达成时，成功总结为选填字段
- 不允许补填历史日期
- 每天 19:00 系统发送未填报提醒（飞书）
- 每天 20:00 后，督导收到 1 条汇总通知（未填报名单）

**状态判定规则：**
- 绿（达成）：当日/当月累计完成率 ≥ 100%
- 红（未达成）：当日/当月累计完成率 < 100%
- 无黄色中间态

**排行榜规则：**
- 统计周期：周一至周日
- 破平规则：完成率相同时以销售额绝对值降序排列
- 月中新入职的店员当月不参与排名
- 冠军风采照上传后长期沿用，直至被新冠军替代或手动更新

**通知合并规则：**
- 所有批量事件（未填报名单、预警名单）合并为 1 条飞书汇总消息，不逐条发送

### 模块划分

**1. 认证与组织模块（Auth & Org）**
- 飞书 OAuth 2.0 授权码流程
- 飞书通讯录 API 拉取组织层级，建立 regions / stores / users 记录
- 角色自动映射规则
- JWT 会话管理
- 组织架构变更自动同步 + 手动触发全量同步

**2. P · 目标与计划模块（Target Management）**
- 多层级目标创建、逐级下发、修改
- 实时校验：超发允许，少发报错；门店→店员严格等额
- 月参考周目标自动推算
- 月 1 日未完成提醒 Cron Job
- 目标下发/修改飞书通知

**3. D · 落地与执行模块（Daily Report）**
- 日报表单：销售额（必填）+ 未达成原因（未达成时必填）+ 成功总结（达成时选填）
- 20:00 截止逻辑（服务端时间判定）
- 19:00 未填报提醒 Cron Job
- 20:00 逾期自动标记 + 督导汇总通知 Cron Job
- 不允许补填历史日期（接口层拦截）

**4. C · 状态与监控模块（Status & Alerts）**
- 完成率实时计算引擎（日/月维度）
- 红/绿状态判定（阈值：100%）
- 连续 2 日未填报预警 Cron Job
- 月完成率低于 60% 预警 Cron Job
- 飞书预警汇总通知

**5. 排行榜模块（Leaderboard）**
- 周度排名计算（完成率 + 销售额两维度）
- 破平规则（销售额绝对值）
- 新入职店员排除逻辑
- 冠军风采照上传与持久化（阿里云 OSS）
- 角色可见性控制（督导只见管辖范围）

**6. A · 复盘与总结模块（Weekly Review）**
- 每周一 Cron Job 为每位督导触发 AI 生成（并行，各自独立）
- Claude API 调用：传入督导管辖范围上周数据，生成包含具体数字的复盘文本
- 督导编辑界面（富文本）+ 发布流程
- 发布后可反复编辑再发布，记录最后发布版本和时间
- 发布触发飞书通知至所有管辖店员

**7. 经验推送模块（Experience Feed）**
- 管理员从成功总结库中挑选内容推送
- FIFO 队列（上限 10 条，新增自动移除最旧）
- 首页 Feed 展示

**8. 通知中心模块（Notification）**
- 统一封装飞书机器人消息 API
- 事件类型：目标下发/修改、19:00 日报提醒、20:00 未填报汇总、连续未填报预警、月完成率预警、复盘发布
- 所有批量通知合并为单条汇总消息

**9. 数据看板模块（Dashboard）**
- 按角色渲染不同数据视图（5 种角色 × 不同维度）
- 核心指标聚合计算（实时）
- 最近 3 个月月度完成率趋势（折线图）
- 下钻能力（总监→片区→督导→门店）

**10. 数据归档模块（Archive）**
- 年底自动归档上一年度数据
- 归档数据只读可查，不影响当年数据
- 跨年度历史查阅入口

### 数据库 Schema（核心表）

```
users
  id, feishu_user_id, name, role (DIRECTOR/REGIONAL_MANAGER/SUPERVISOR/STAFF/ADMIN),
  region_id (→ regions, nullable), store_id (→ stores, nullable),
  supervisor_id (→ users, nullable), avatar_url, profile_photo_url,
  joined_at, created_at, updated_at

regions
  id, name, manager_id (→ users), created_at

stores
  id, name, region_id (→ regions), supervisor_id (→ users), created_at

targets
  id, assignee_id (→ users), store_id (→ stores, nullable),
  assigner_id (→ users),
  period_type (MONTHLY/WEEKLY_REF/DAILY_REF), period_start, period_end,
  amount, is_archived,
  created_at, updated_at

daily_reports
  id, user_id (→ users), store_id (→ stores),
  report_date, actual_amount, is_achieved,
  failure_reason (text, nullable), success_summary (text, nullable),
  is_late (bool), submitted_at, created_at

weekly_reviews
  id, supervisor_id (→ users), week_start, week_end,
  ai_content (text), edited_content (text, nullable),
  status (DRAFT/PUBLISHED), last_published_at, created_at, updated_at

experience_posts
  id, source_report_id (→ daily_reports, nullable),
  pushed_by (→ users), content (text),
  author_name (varchar), store_name (varchar),
  created_at  -- FIFO 排序依据，保留最新 10 条

champion_photos
  id, user_id (→ users), week_start,
  photo_url (varchar), uploaded_by (→ users), created_at

notification_logs
  id, event_type (varchar), recipient_id (→ users),
  feishu_message_id (varchar), sent_at, payload (jsonb)
```

### 关键 API 端点

| 方法 | 路径 | 说明 |
|------|------|------|
| `POST` | `/api/auth/feishu/callback` | 飞书 OAuth 回调，创建会话 |
| `POST` | `/api/auth/sync-org` | 触发飞书组织架构同步（管理员） |
| `GET` | `/api/targets` | 获取当前用户目标（含层级上下文） |
| `POST` | `/api/targets` | 创建并下发目标 |

## Testing Decisions

**测试原则：** 只测外部行为（输入→输出），不测实现细节。核心逻辑必须在数据库层面验证状态变化，不 Mock 业务规则。

| 模块 | 测试重点 |
|------|----------|
| 目标管理 | 超发允许（通过）/ 少发拦截（报错）；门店→店员严格等额校验；修改目标后通知正确触发 |
| 日报填报 | 未达成时跳过原因字段被拦截；20:00 后提交被标记逾期；不允许补填历史日期 |
| 状态监控 | 完成率边界值（99.9% 为红，100% 为绿）；连续 2 日未填报预警正确触发 |
| 数据权限 | 店员无法获取其他店员数据；督导无法获取非管辖范围数据；只读角色无法执行写操作 |
| 排行榜 | 完成率相同时销售额绝对值破平排序正确；新入职店员不出现在排行榜 |
| 经验推送 | 第 11 条推送后最旧一条被移除，列表始终 ≤ 10 条 |
| 通知合并 | 多个未填报事件合并为 1 条飞书消息发出 |
| 年度归档 | 归档后历史数据可读不可写；当年数据不受影响 |

## Further Notes

- 飞书组织架构同步需要企业管理员提前为飞书应用授权通讯录读取权限，上线前需提前申请审批
- AI 周复盘使用 Claude API，需确认阿里云部署环境的网络可访问性；备选方案为通义千问
- 冠军风采照存储使用阿里云 OSS，需提前创建 Bucket 并配置 CDN 加速
- 「连续 2 日未填报」预警一期按自然日计算，不区分节假日；节假日豁免逻辑列为二期
- 19:00 / 20:00 定时任务基于服务器时间（UTC+8），部署时须配置正确时区
- 首次上线第一个月无历史对比数据，趋势图从第 2 个月起有意义，前端需处理空数据态
- 目标修改后若下级已做子目标分配，需在界面提示下级重新检查分配是否仍满足规则
